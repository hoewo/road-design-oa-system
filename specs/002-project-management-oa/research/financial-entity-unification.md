# 财务实体统一设计方案

**设计日期**: 2025-01-28  
**设计目标**: 统一所有财务相关实体，简化数据模型，提高财务统计的一致性

## 当前财务相关实体分析

### 现有财务相关实体

1. **经营奖金**
   - 字段：发放人员、发放金额、发放时间
   - 方向：支出
   - 类型：奖金

2. **生产奖金**
   - 字段：发放人员、发放金额、发放时间
   - 方向：支出
   - 类型：奖金

3. **甲方支付**
   - 字段：支付时间、金额
   - 方向：收入
   - 类型：收款

4. **我方开票**
   - 字段：开票时间、金额、发票文件
   - 方向：收入（开票阶段）
   - 类型：开票

5. **专家费支付**（在招投标信息中）
   - 字段：现金/转账、金额、人员
   - 方向：支出
   - 类型：专家费

6. **生产成本**
   - 字段：打车（时间、里程、金额）、住宿（时间、金额）、公共交通（时间、金额）、发票
   - 方向：支出
   - 类型：成本

7. **对外委托支付**（在对外委托中）
   - 字段：金额、时间、索要发票
   - 方向：支出
   - 类型：委托支付

## 统一财务记录实体设计

### 核心实体：财务记录

**财务记录**：代表所有与金额进账和出账相关的财务信息，包括奖金、成本、支付、开票等。

#### 实体字段设计

**基础字段**：
- ID（主键）
- 项目ID（外键，关联到项目）
- 财务类型（枚举：奖金、成本、支付、开票、专家费、委托支付等）
- 方向（枚举：收入、支出）
- 金额（decimal，必填）
- 发生时间（date，必填）
- 创建时间（timestamp）
- 更新时间（timestamp）
- 创建者ID（外键，关联到用户）
- 更新者ID（外键，关联到用户）
- 备注（string，可选）

**类型特定字段**（根据财务类型使用不同字段）：

1. **奖金类型**（财务类型 = 奖金）
   - 奖金类别（枚举：经营奖金、生产奖金）
   - 发放人员ID（外键，必填，关联到用户ID）
   - 说明：发放人员必须是项目中的用户，用户是否有账号取决于管理员的配置

2. **成本类型**（财务类型 = 成本）
   - 成本类别（枚举：打车、住宿、公共交通）
   - 里程（decimal，可选，仅打车类型）
   - 发票文件ID（外键，可选，关联到文件）

3. **甲方支付类型**（财务类型 = 甲方支付）
   - 甲方ID（外键，必填，关联到甲方）
   - 说明：甲方支付后，我方需要向甲方提供发票并记录（通过关联的我方开票记录）

4. **我方开票类型**（财务类型 = 我方开票）
   - 甲方ID（外键，必填，关联到甲方）
   - 发票文件ID（外键，必填，关联到文件）
   - 关联甲方支付记录ID（外键，可选，关联到财务记录，表示开票对应的甲方支付）
   - 说明：我方向甲方开票，是甲方支付的后续阶段

5. **专家费类型**（财务类型 = 专家费）
   - 支付方式（枚举：现金、转账）
   - 专家人员（string，必填，专家姓名）

6. **委托支付类型**（财务类型 = 委托支付）
   - 委托类型（枚举：个人、单位）
   - 委托方名称（string，必填）
   - 委托方评分（decimal，可选，对委托方的评分）
   - 说明：对外支付后，对方需要提供发票，我方将发票上传（通过关联的对方开票记录）

7. **对方开票类型**（财务类型 = 对方开票）
   - 委托方名称（string，必填）
   - 发票文件ID（外键，必填，关联到文件，对方提供的发票）
   - 关联委托支付记录ID（外键，可选，关联到财务记录，表示开票对应的委托支付）
   - 说明：委托方给我方开票，是委托支付的后续阶段

**扩展字段**（用于未来扩展）：
- 扩展数据（JSON，可选，存储类型特定的额外信息）

### 财务类型枚举定义

```
财务类型枚举：
- 奖金：经营奖金、生产奖金（通过奖金类别字段区分）
- 成本：打车、住宿、公共交通（通过成本类别字段区分）
- 甲方支付：甲方支付给我方（收入）
- 我方开票：我方向甲方开票（收入，关联甲方支付）
- 专家费：专家费支付（支出）
- 委托支付：对外委托支付（支出）
- 对方开票：委托方给我方开票（收入，关联委托支付）
```

**说明**：
- 奖金和成本通过类别字段区分子类型
- 支付和开票分为不同的财务类型，便于区分业务场景
- 我方开票和对方开票是不同的业务场景（我方开票给甲方 vs 委托方开票给我方）

### 方向枚举定义

```
方向枚举：
- 收入：所有进账（甲方支付、我方开票、对方开票）
- 支出：所有出账（奖金、成本、专家费、委托支付）
```

**说明**：
- 甲方支付：方向=收入（甲方支付给我方）
- 我方开票：方向=收入（开票记录，虽然不产生实际收入，但作为财务流程的一部分）
- 对方开票：方向=收入（收到委托方的发票，作为成本凭证）

## 实体关系设计

### 财务记录与项目的关联
- 关系：多对一（一个项目可以有多个财务记录）
- 外键：财务记录.项目ID → 项目.ID

### 财务记录与文件的关联
- 关系：多对一（一个财务记录可以关联一个发票文件）
- 外键：财务记录.发票文件ID → 文件.ID

### 财务记录之间的关联
- 关系：一对多（一个支付记录可以对应多个开票记录）
- 外键：财务记录.关联支付记录ID → 财务记录.ID

### 财务记录与用户的关联（创建者、更新者）
- 关系：多对一（创建者、更新者）
- 外键：财务记录.创建者ID → 用户.ID
- 外键：财务记录.更新者ID → 用户.ID

### 财务记录与用户的关联
- 关系：多对一（奖金发放人员）
- 外键：财务记录.发放人员ID → 用户.ID
- 说明：发放人员必须是项目中的用户，用户是否有账号取决于管理员的配置

### 财务记录与甲方的关联
- 关系：多对一（甲方支付、我方开票）
- 外键：财务记录.甲方ID → 甲方.ID
- 说明：甲方支付和我方开票都关联到甲方

### 财务记录之间的关联
- 关系：一对多（一个支付记录可以对应多个开票记录）
- 外键：财务记录.关联甲方支付记录ID → 财务记录.ID（我方开票关联甲方支付）
- 外键：财务记录.关联委托支付记录ID → 财务记录.ID（对方开票关联委托支付）
- 说明：支持支付和开票的关联，跟踪完整的业务流程

## 业务场景映射

### 场景1：甲方支付和我方开票（同一件事的不同阶段）

**原设计**：
- 甲方支付：支付时间、金额
- 我方开票：开票时间、金额、发票文件

**新设计**：
- 财务记录1（甲方支付）：
  - 财务类型：甲方支付
  - 方向：收入
  - 甲方ID：关联到甲方
  - 金额、时间
- 财务记录2（我方开票）：
  - 财务类型：我方开票
  - 方向：收入
  - 甲方ID：关联到甲方
  - 金额、时间、发票文件ID
  - 关联甲方支付记录ID：指向财务记录1
- **说明**：甲方支付后，我方需要向甲方提供发票并记录，两个记录通过关联字段连接

### 场景2：对外委托支付和委托方开票

**原设计**：
- 对外委托：给委托方支付（金额、时间、索要发票）

**新设计**：
- 财务记录1（委托支付）：
  - 财务类型：委托支付
  - 方向：支出
  - 委托类型：个人/单位
  - 委托方名称：委托方名称
  - 委托方评分：对委托方的评分
  - 金额、时间
- 财务记录2（对方开票，如果对方提供发票）：
  - 财务类型：对方开票
  - 方向：收入（收到委托方发票，作为成本凭证）
  - 委托方名称：委托方名称
  - 金额、时间、发票文件ID（对方提供的发票）
  - 关联委托支付记录ID：指向财务记录1
- **说明**：对外支付后，对方需要提供发票，我方将发票上传，两个记录通过关联字段连接

### 场景3：经营奖金和生产奖金

**原设计**：
- 经营奖金：发放人员、发放金额、发放时间
- 生产奖金：发放人员、发放金额、发放时间

**新设计**：
- 财务记录（奖金类型）：
  - 财务类型：奖金
  - 方向：支出
  - 奖金类别：经营奖金/生产奖金
  - 发放人员ID：关联到用户ID
  - 发放金额、发生时间
- **说明**：发放人员必须是项目中的用户，用户是否有账号取决于管理员的配置

### 场景4：生产成本

**原设计**：
- 生产成本：打车（时间、里程、金额）、住宿（时间、金额）、公共交通（时间、金额）、发票

**新设计**：
- 财务记录（成本类型）：
  - 财务类型：成本
  - 方向：支出
  - 成本类别：打车/住宿/公共交通
  - 金额、时间
  - 里程（仅打车类型）
  - 发票文件ID

### 场景5：专家费支付

**原设计**：
- 招投标信息：专家费支付信息（现金/转账、金额、人员）

**新设计**：
- 财务记录（专家费类型）：
  - 财务类型：专家费

  - 金额、时间

## 优势分析

### 1. 数据模型简化
- 从7个财务相关实体统一为1个财务记录实体
- 减少数据模型复杂度
- 统一财务统计逻辑

### 2. 业务逻辑清晰
- 通过财务类型和方向字段清晰区分不同业务场景
- 支持同一业务流程的不同阶段（支付→开票）
- 便于扩展新的财务类型

### 3. 统计计算统一
- 所有财务统计基于统一的财务记录实体
- 收入统计：方向=收入的所有记录
- 支出统计：方向=支出的所有记录
- 按财务类型分组统计

### 4. 关联关系明确
- 支付和开票的关联通过"关联支付记录ID"字段
- 支持一对多关系（一个支付可以对应多个开票）

## 设计优化说明

### 根据业务反馈的优化

1. **奖金发放人员关联优化**
   - ✅ 发放人员必须关联到项目中的用户
   - ✅ 通过发放人员ID关联到用户实体
   - ✅ 用户是否有账号取决于管理员的配置

2. **支付类型简化**
   - ✅ 支付类型明确为"甲方支付"，无支付方式区别
   - ✅ 只关联甲方ID，不关联其他对象
   - ✅ 甲方支付后，我方需要向甲方提供发票并记录

3. **委托支付完善**
   - ✅ 委托类型：个人或单位
   - ✅ 委托方评分：对委托方的评分
   - ✅ 对外支付后，对方需要提供发票，我方将发票上传
   - ✅ 通过"对方开票"类型记录委托方给我方的发票

4. **开票类型区分**
   - ✅ "我方开票"：我方向甲方开票（收入，关联甲方支付）
   - ✅ "对方开票"：委托方给我方开票（收入，关联委托支付）
   - ✅ 两个开票类型业务场景不同，需要区分

## 注意事项

1. **向后兼容**：需要考虑现有数据的迁移和兼容
2. **业务规则**：不同类型可能有不同的业务规则，需要在应用层处理
3. **查询性能**：统一实体可能增加查询复杂度，需要合理设计索引
4. **扩展性**：通过扩展数据字段支持未来可能的扩展需求
5. **人员关联**：奖金发放人员必须关联到项目中的经营人员或生产人员，不能是外部人员
6. **开票方向**：虽然开票记录方向为"收入"，但实际业务含义不同（我方开票是收入凭证，对方开票是成本凭证）
