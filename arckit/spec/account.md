# 账号管理功能规格

**模块**: project-management  
**状态**: 🟢 已实现  
**最后更新**: 2026-03-17

## 概述

**关联需求**：[登录页品牌化（公司全称、Logo、布局与可理解性）](./login-page-branding.md) — 描述登录入口的产品期望，与本文账号/认证能力配合；若与任务层描述冲突，以 `login-page-branding.md` 中待裁决表及更新后的本文为准。

系统管理员能够预设用户账号、编辑用户信息，预设账号的用户可以登录系统。系统采用基于 NebulaAuth 的验证码登录方式，不提供公开注册功能，所有账号必须由系统管理员预先创建和管理。

## 用户故事

### US-01 账号管理 (Priority: P1)

**场景描述**：系统管理员能够预设用户账号、编辑用户信息，预设账号的用户可以登录系统。系统需要提供安全的身份认证机制，确保只有授权用户才能访问系统功能。系统不提供公开注册功能，所有账号必须由系统管理员预先创建和管理。

**验收场景**：

1. **Given** 系统管理员需要创建新用户账号，**When** 管理员在用户管理界面创建账号（邮箱或手机号、用户名等信息），**Then** 系统创建账号并允许该用户使用验证码登录
2. **Given** 用户已有预设账号，**When** 用户使用邮箱或手机号+验证码登录，**Then** 系统验证身份并允许访问
3. **Given** 用户尝试登录，**When** 用户输入错误的验证码，**Then** 系统拒绝访问并提示错误信息
4. **Given** 用户尝试访问注册功能，**When** 用户尝试注册新账号，**Then** 系统不提供注册功能，提示用户联系管理员创建账号
5. **Given** 系统管理员需要修改用户信息，**When** 管理员在用户管理界面编辑用户信息（用户名、真实姓名、邮箱、手机号、部门、角色、激活状态等所有信息），**Then** 系统更新用户信息并保存修改

## 功能需求

- **FR-001**: 系统必须提供用户账号登录、登出功能，确保安全的身份认证。系统采用基于 NebulaAuth 的验证码登录方式（支持邮箱和手机号），系统不提供公开注册功能。系统管理员可以预设用户账号、编辑用户信息（包括用户名、真实姓名、邮箱、手机号、部门、角色、激活状态等所有信息）。如果用户是系统管理员，角色不能修改
- **FR-002**: 系统必须支持用户权限管理，区分项目管理员、经营负责人、生产负责人、财务人员等不同角色。用户角色支持多选，一个用户可以同时拥有多个角色（如：项目管理员+经营负责人）。系统管理员具备所有角色类型的权限，其他角色只有自己角色的权限

## 边界情况

- **EC-001**: 默认 admin 账号的初始密码设置：系统初始化时创建默认 admin 账号，初始密码固定（如"admin123"），首次登录时提示但不强制修改密码
- **EC-002**: admin 账号删除或禁用：系统不允许删除或禁用 admin 账号，系统只有一个 admin 账号
- **EC-003**: 账号注销：系统不允许账号注销功能
- **EC-036**: 账号注册功能：系统不提供公开注册功能，所有账号必须由系统管理员预先创建。用户无法自行注册账号
- **EC-037**: 管理员预设账号：系统管理员可以在用户管理界面创建新账号，设置用户名、邮箱或手机号、角色等信息。预设账号的用户可以使用邮箱或手机号+验证码登录系统
- **EC-038**: 用户信息编辑：系统管理员可以编辑用户的所有信息，包括用户名、真实姓名、邮箱、手机号、部门、角色、激活状态等。系统不存储密码，登录通过 NebulaAuth 的验证码方式。如果用户是系统管理员，角色不能修改
- **EC-039**: 编辑用户权限验证：只有系统管理员可以编辑用户信息，非管理员用户无法编辑其他用户信息
- **EC-040**: 用户角色多选：用户角色支持多选，一个用户可以同时拥有多个角色（如：项目管理员+经营负责人）。角色字段使用数组/列表类型存储。**注意**：虽然用户角色中可能有"经营负责人"和"生产负责人"角色，但这些角色仅用于在配置项目负责人时进行用户过滤，实际的经营负责人和生产负责人是项目级别的，通过项目的 BusinessManagerID 和 ProductionManagerID 字段确定
- **EC-041**: 系统管理员权限：系统管理员具备所有角色类型的权限，其他角色只有自己角色的权限
- **EC-042**: 项目配置用户角色限制：项目中关于人的配置，如果是选择的系统已有用户，应该只能选择有对应角色权限的用户
- **EC-044**: 系统管理员角色保护：如果用户是系统管理员，系统不允许修改其角色，防止误操作导致系统管理员权限丢失
- **EC-045**: 用户角色和项目成员角色的权限关系：权限检查同时考虑用户角色（UserRole）和项目成员角色（MemberRole）。用户角色决定全局权限，项目成员角色决定项目内权限。两者取并集（任一满足即可）。**重要**：经营负责人和生产负责人是项目级别的角色，不存在全局的经营负责人或生产负责人角色
- **EC-046**: 配置项目成员的权限检查：配置项目成员时，检查：1) 用户角色是否为项目管理员或系统管理员（全局权限）；2) 如果是特定项目的经营负责人（通过 Project.BusinessManagerID 判断），可以配置经营参与人；3) 如果是特定项目的生产负责人（通过 Project.ProductionManagerID 判断），可以配置生产人员。系统管理员和项目管理员可以配置所有类型的项目成员
- **EC-047**: 选择项目成员时的用户过滤规则：配置项目成员（MemberRole）时，可以选择所有用户，不需要角色过滤。但配置项目负责人（经营负责人、生产负责人）时，需要根据角色过滤
- **EC-048**: 用户角色权限等级：权限等级从高到低：系统管理员 > 项目管理员/财务人员 > 普通成员

## 权限规则

系统采用双重权限检查机制（详见 overview.md 权限总规则）：

- 用户角色（UserRole）决定全局权限
- 项目成员角色（MemberRole）决定项目内权限
- 两者取并集（任一满足即可）

**角色保护**：
- 系统管理员角色不可修改
- 只有项目管理员角色才能创建项目

## 关键实体

- **用户（User）**：统一账号实体，包含用户名、邮箱、真实姓名、联系电话、部门、角色（数组）、激活状态等字段
